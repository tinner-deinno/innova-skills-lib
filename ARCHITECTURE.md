## Design Principles

1. **Single source of truth** — every skill lives once in this repo as a `SKILL.md` with YAML frontmatter.
2. **Runtime neutrality** — skill bodies are plain Markdown; runtime-specific formatting is produced by adapters.
3. **Append-only / archive-only** — skills are never deleted; deprecated skills move to `deprecated/` inside their category.
4. **Private by containment** — `private/` skills are never scanned, manifested, or adapted.
5. **Generated adapters** — files under `adapters/` are produced by tooling and must not be hand-edited.

## Layered Architecture

### 1. Skill Source Layer (`core/`, `ECC/`, `gov/`, `finance/`, `9arm/`)

- Each skill is a directory: `<category>/<skill-name>/SKILL.md`.
- Optional supporting files (scripts, examples, references) may live alongside `SKILL.md`.
- `SKILL.md` must begin with YAML frontmatter:

```yaml
---
name: my-skill
description: One-sentence description used in manifests and plugin registries.
---
```

### 2. Registry Layer (`manifests/`)

- `skills_manifest.json` — machine-readable JSON registry of every distributable skill.
- `skills_index.md` — human-readable markdown index grouped by category.
- Both are generated by `tools/register_skills.py`.

### 3. Adapter Layer (`adapters/`)

- `generate_adapters.py` reads the manifest and canonical `SKILL.md` files.
- It strips YAML frontmatter and writes runtime-specific adapter files.
- Today only `adapters/claude/` is produced; `copilot/`, `codex/`, `hermes/` are planned.

### 4. Tooling Layer (`tools/`)

| Tool | Responsibility | Inputs | Outputs |
|---|---|---|---|
| `register_skills.py` | Scan source skills, build registry/index | `~/.claude/skills/` | `manifests/skills_manifest.json`, `skills_index.md` |
| `generate_adapters.py` | Generate runtime adapter files | manifest + `~/.claude/skills/` | `adapters/<runtime>/*.md` |
| `sync_to_project.ps1` | Install global skills into a project | `~/.claude/skills/`, target project path | `<project>/.claude/skills/` |

### 5. Distribution Layer (external)

- `~/.claude/skills/` — global Claude Code skills directory (live, active).
- Project-local `.claude/skills/` — per-repo skill overrides created by `sync_to_project.ps1`.

## Adapter Format (Claude)

Generated files are Markdown with HTML metadata comments and the skill body:

```markdown
<!-- skill-id: learn -->
<!-- source-path: learn -->
<!-- source-file: C:/Users/.../.claude/skills/learn/SKILL.md -->
<!-- runtime: claude -->

# /learn - Deep Dive Learning Pattern
...
```

The frontmatter is stripped because Claude Code skills are loaded as Markdown instruction files, not YAML-frontmatter documents.

## Tooling Behavior

### `register_skills.py`

1. Walks `~/.claude/skills/` recursively.
2. Finds every directory containing `SKILL.md`.
3. Parses YAML frontmatter (simple regex parser, no PyYAML dependency).
4. Derives `id`, `category`, `description` (fallback to first body line).
5. Writes `manifests/skills_manifest.json` and `skills_index.md`.

### `generate_adapters.py`

1. Loads `manifests/skills_manifest.json`.
2. Supports both `{skills:[...]}` and raw `[...]` manifest shapes.
3. Skips entries whose `runtimes` does not include `claude`.
4. Reads source `SKILL.md` from `~/.claude/skills/<path>/SKILL.md`.
5. Strips frontmatter and writes `adapters/claude/<skill-id>.md`.

### `sync_to_project.ps1`

1. Iterates top-level directories in `~/.claude/skills/`.
2. Creates a directory junction (default) or full copy in `<project>/.claude/skills/`.
3. Flattens the special `ecc/` subtree into `ecc-<subname>` entries to match innova-bot's loader convention.
4. Supports `-DryRun`, `-Force`, and `-Mode copy`.

---

## Boundaries

- `private/` is excluded from all tooling by `.gitignore` and convention.
- `adapters/` is generated output — never hand-edit.
- `manifests/skills_manifest.json` and `skills_index.md` are generated — edit skills, not the manifest.
- `deprecated/` inside categories archives retired skills.
